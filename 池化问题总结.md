总体原因可以这样总结：

PR 里把 DSV4 compressed attention 的 `positions_cpu` 改成了从 NPU tensor 现场取：

```python
positions_cpu=self.positions.cpu()
```

而 KV Pool 开启后，`pool_worker.py` 会先对 KV cache buffer 调：

```python
m_store.register_buffer(...)
```

这个注册最终会进 Mooncake TE/Ascend transport，把 NPU KV cache memory 注册给外部传输引擎使用。注册本身能正常返回，但它会改变当前进程里的 Ascend runtime/TE 内存注册状态。

后面 DSV4 构造 attention metadata 时，`self.positions.cpu()` 会触发一次 NPU -> CPU 同步拷贝。也就是说 Python 主线程在 metadata 阶段开始等待 NPU stream drain。这个同步点和 TE 已经注册过 NPU memory 的 runtime 状态组合起来，就会触发 hang。表现上可能卡在 `dsa_v1.py` 的 `prefill_slot_mapping`、`.item()`、`positions_cpu` 相关位置，但本质都是同一类问题：metadata 里的 Python 控制流依赖了 NPU 异步结果。

正常分支为什么 OK：它不是从 NPU tensor 反向 `.cpu()` 拿 positions，而是在 `_prepare_inputs` 阶段已经用 CPU buffer 算好了 positions，再 copy 到 NPU 给模型用。CPU 侧 metadata 直接读 CPU staging buffer，不需要等待 NPU。

所以有效修复是：

```python
positions_cpu=(
    self._positions_cpu_buf[:num_tokens]
    if self.use_compress
    else None
)
```

这样 DSV4 metadata 使用的 `positions_cpu` 来自 CPU staging，`dsa_v1.py` 里的 decode/prefill 切片、长度计算、compressed token start/end 等 Python 控制逻辑都不再触发 NPU 同步。

一句话：**register_buffer 不是挂点，但它让 Ascend runtime 进入 TE 注册态；`self.positions.cpu()` 是后续第一个危险同步点。两者组合导致 graph capture/metadata build 阶段 hang。修复就是让 DSV4 metadata 的 CPU 控制量只来自 CPU buffer，避免在注册后从 NPU tensor 做 `.cpu()`/`.item()`/slice 这类 host sync。**

定位结果

  "No available shared memory broadcast block found in 60 seconds" 来自 vLLM shm_broadcast.py:81-88（long_wait_time_msg，在 acquire_write/acquire_read 等待 ring buffer 满 60s 时打印）。这说明
  driver/scheduler 向 worker 广播的 ring buffer 卡住了 —— 通常是 driver 主进程在某个 NPU sync 阶段 hang 住，TP worker（reader）等不到 broadcast 写入而开始刷 warning。

  为什么 ASCEND_BUFFER_POOL=4:8 修好了，但 ASCEND_ENABLE_USE_FABRIC_MEM=1 又复现

  mooncake_backend.py:39/77 在 fabric_mem 开关打开时确实跳过了 global_te.register_buffer()。但它走的是 store.setup(..., local_buffer_size=0, ...) 的另一路径——即直接用统一显存地址（UMA fabric
  memory）方案，mooncake 内部仍会把 NPU 显存做"地址直通映射"，这同样会让 Ascend runtime 进入一个"显存外部可见"的状态。在这个状态下，任何 NPU→CPU 同步点都可能 hang，跟上次 TE 注册态同质。

  所以问题没变：dsa_v1 metadata 构造路径里还有残留的 NPU host sync，上次的修复只清掉了 decode 路径（通过 positions_cpu），prefill 路径还没动。

  仍未修掉的残留同步点（都在 vllm_ascend/attention/dsa_v1.py 的 build_prefill_metadata 里）

  1. dsa_v1.py:598
  input_positions = common_attn_metadata.positions[: self.num_actual_tokens].long()
  1. prefill 路径仍然从 NPU positions 取，没用上次新加的 positions_cpu。
  2. dsa_v1.py:680-689（隐式 sync，最危险）
  decode_input_positions = input_positions[:tokens_start]  # NPU tensor
  compressed_tokens_start, compressed_tokens_end = _get_compressed_decode_token_start_and_end(
      decode_input_positions, self.compressor_ratio)
  2. _get_compressed_decode_token_start_and_end 内部 mask.sum() 返回的是 NPU 0-dim tensor（没 .item()），看上去没显式 sync。
  3. dsa_v1.py:698-700（隐式 sync）
  prefill_slot_mapping = self.slot_mapping[
      compressed_tokens_start : compressed_tokens_end + compressed_tokens_start
  ]
  3. compressed_tokens_start 是 NPU 0-dim tensor，Python slice 会调用 __index__ → 隐式 .item() → NPU→CPU 同步。这就是 prefill_slot_mapping 那个 hang 点。
  4. dsa_v1.py:727 / 752 / 779
  max_seqlen_kv=self.seq_lens[reqs_start:].max(),
  4. 0-dim NPU tensor 传进 custom op，C++ 侧若 tensor.item() 取标量也会 sync。
  5. dsa_v1.py:804-805（显式 .item()，最直接）
  max_seqlen_q=seq_lens_q.max().item(),
  max_seqlen_k=self.seq_lens[reqs_start:].max().item(),
  5. 直接 NPU→CPU 同步。

  build_decode_metadata（856 行起）已经全部走 CPU（positions_cpu、_seq_lens_cpu），干净。所以为啥之前 ASCEND_BUFFER_POOL=4:8 不再 hang —— warmup/dummy_run 阶段 attn_state 是
  DecodeOnly/SpecDecoding（model_runner_v1.py:3125-3131），不会进 prefill builder；真实推理的第一个 prefill 请求又恰好不踩在 TE 已注册显存的危险窗口里。但 fabric_mem 是整个进程生命周期都处于 UMA
  状态，所以第一个真正带 prefill 的请求到来时就在 698-700/804-805 上 hang，driver hang → worker shm_broadcast 60s timeout 刷屏。
  
  建议的修复方向（保持和之前修复同构）

  把 prefill builder 改成像 decode builder 一样走 positions_cpu 和 _seq_lens_cpu：

  - 把 598 行改成读 common_attn_metadata.positions_cpu[: self.num_actual_tokens].long()（同时保留 NPU input_positions 给 RoPE 等用 cos/sin 的地方），并在 680 行用 CPU 版本去算 decode_input_positions_cpu 用于
   mask；
  - _get_compressed_decode_token_start_and_end 内部对 CPU tensor 做 mask.sum().item() → 返回 Python int，698 行 slice 就不会再触发 NPU sync；
  - 804/805 改用 _seq_lens_cpu[reqs_start:].max().item()（已经在 602-608 取过了，可以复用），seq_lens_q.max().item() 改成 (prefill_query_start_loc_cpu[1:] - prefill_query_start_loc_cpu[:-1]).max().item()；
  - 727/752/779 的 self.seq_lens[reqs_start:].max() 如果 C++ kernel 支持 Python int，就也改成 .item() 后传值；否则保留 device tensor 但确认 kernel 不会 host-sync。
