总体原因可以这样总结：

PR 里把 DSV4 compressed attention 的 `positions_cpu` 改成了从 NPU tensor 现场取：

```python
positions_cpu=self.positions.cpu()
```

而 KV Pool 开启后，`pool_worker.py` 会先对 KV cache buffer 调：

```python
m_store.register_buffer(...)
```

这个注册最终会进 Mooncake TE/Ascend transport，把 NPU KV cache memory 注册给外部传输引擎使用。注册本身能正常返回，但它会改变当前进程里的 Ascend runtime/TE 内存注册状态。

后面 DSV4 构造 attention metadata 时，`self.positions.cpu()` 会触发一次 NPU -> CPU 同步拷贝。也就是说 Python 主线程在 metadata 阶段开始等待 NPU stream drain。这个同步点和 TE 已经注册过 NPU memory 的 runtime 状态组合起来，就会触发 hang。表现上可能卡在 `dsa_v1.py` 的 `prefill_slot_mapping`、`.item()`、`positions_cpu` 相关位置，但本质都是同一类问题：metadata 里的 Python 控制流依赖了 NPU 异步结果。

正常分支为什么 OK：它不是从 NPU tensor 反向 `.cpu()` 拿 positions，而是在 `_prepare_inputs` 阶段已经用 CPU buffer 算好了 positions，再 copy 到 NPU 给模型用。CPU 侧 metadata 直接读 CPU staging buffer，不需要等待 NPU。

所以有效修复是：

```python
positions_cpu=(
    self._positions_cpu_buf[:num_tokens]
    if self.use_compress
    else None
)
```

这样 DSV4 metadata 使用的 `positions_cpu` 来自 CPU staging，`dsa_v1.py` 里的 decode/prefill 切片、长度计算、compressed token start/end 等 Python 控制逻辑都不再触发 NPU 同步。

一句话：**register_buffer 不是挂点，但它让 Ascend runtime 进入 TE 注册态；`self.positions.cpu()` 是后续第一个危险同步点。两者组合导致 graph capture/metadata build 阶段 hang。修复就是让 DSV4 metadata 的 CPU 控制量只来自 CPU buffer，避免在注册后从 NPU tensor 做 `.cpu()`/`.item()`/slice 这类 host sync。**
